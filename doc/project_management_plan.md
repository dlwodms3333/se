# 1. 서론

## 1.1 문서의 목적 및 범위

본 문서는 조직 내 협업을 위한 클라우드 파일 공유 시스템인 **Mini Drive** 프로젝트를 수행하기 위한 프로젝트 관리 계획서이다.

본 문서에서는 프로젝트 개요, 개발 계획, 팀 구성, 품질 관리, 개발 환경, 산출물, 리스크 관리 내용을 정리한다.

---

## 1.2 프로젝트 개요

### 1.2.1 프로젝트 정의

Mini Drive는 조직 내부 사용자를 위한 클라우드 기반 파일 관리 및 공유 시스템이다.

현재 조직 내 파일은 이메일, 메신저, 개인 클라우드 등을 통해 분산되어 공유되는 경우가 많다. 이로 인해 최신 파일 확인, 파일 검색, 외부 공유 시 보안 관리에 어려움이 발생할 수 있다.

Mini Drive는 파일을 중앙에서 관리하고, 업로드, 다운로드, 공유, 검색, 버전 관리, 권한 제어 기능을 제공하여 이러한 문제를 해결하는 것을 목표로 한다.

### 1.2.2 주요 기능 설명

| ID | 기능명 | 설명 |
|---|---|---|
| F01 | 파일 업로드/다운로드 | 사용자가 업무 파일을 업로드하고 다시 다운로드할 수 있다. |
| F02 | 폴더 관리 | 폴더 생성, 이름 변경, 삭제, 파일 이동을 통해 파일을 정리할 수 있다. |
| F03 | 파일 공유 | 특정 사용자에게 파일을 공유하고 보기, 수정, 댓글 권한을 설정할 수 있다. |
| F04 | 외부 링크 공유 | 외부 협력 업체와 링크를 통해 파일을 공유할 수 있다. |
| F05 | 파일 검색 | 파일 이름, 날짜, 사용자, 파일 유형 등을 기준으로 검색할 수 있다. |
| F06 | 파일 버전 관리 | 이전 버전을 확인, 다운로드, 복원할 수 있다. |
| F07 | 사용자 계정 관리 | 로그인 및 관리자 계정 관리를 지원한다. |
| F08 | 보안 관리 | 권한 기반 접근 제어와 공유 링크 만료 기간 설정을 고려한다. |

## 1.3 프로젝트 목표

본 프로젝트의 목표는 조직 내부 사용자가 파일을 안전하고 편리하게 관리할 수 있는 클라우드 파일 공유 시스템을 개발하는 것이다.

구체적인 목표는 다음과 같다.

- 조직 내 파일을 중앙에서 관리한다.
- 파일 업로드, 다운로드, 폴더 관리 기능을 제공한다.
- 사용자 간 파일 공유와 권한 설정을 지원한다.
- 외부 사용자와도 안전하게 파일을 공유할 수 있도록 한다.
- 파일 검색과 버전 관리 기능을 제공한다.
- 향후 기능 확장을 고려한 구조로 설계한다.

# 2. 개발 계획

## 2.1 개발 절차 모형

본 프로젝트는 **Scrum 기반 애자일 개발 방식**을 적용한다.

Mini Drive는 파일 관리, 공유, 검색, 권한 관리 등 여러 기능으로 구성되어 있으므로 기능을 작은 단위로 나누어 반복적으로 개발하고 검토하는 방식이 적합하다.

Scrum 방식을 사용하면 요구사항 변경에 비교적 유연하게 대응할 수 있고, Sprint 단위로 결과물을 확인하며 개선할 수 있다.

# 4. 품질 관리

## 4.1 팀 미팅 및 의사소통 계획

프로젝트 진행 중에는 작업 상황을 주기적으로 확인하고, 회의 방식과 의사소통 도구를 구분하여 사용한다.  
빠른 소통은 메신저를 활용하고, 추적이 필요한 작업 내용과 결정 사항은 GitHub Issue 또는 문서에 기록한다.

| 구분 | 사용 방식 및 도구 | 사용 목적 |
|---|---|---|
| 오프라인 회의 | 대면 회의 | 프로젝트 초기 계획 수립, 주요 기능 범위 결정, 최종 산출물 검토 |
| 온라인 회의 | Zoom, Google Meet, Discord 등 | 원격 회의, 화면 공유를 통한 문서 및 GitHub Repository 검토 |
| 메신저 소통 | 카카오톡 등 | 일정 조율, 간단한 질문, 긴급 공지, 진행 상황 확인 |
| 코드 및 문서 관리 | GitHub Repository | 소스 코드, 문서 산출물, README 관리 |
| 작업 항목 관리 | GitHub Issue | 기능 추가, 오류 수정, 문서 수정 요청, 일정 지연 사항 관리 |
| 문서 공유 | Markdown 문서, GitHub `doc/` 폴더 | 프로젝트 산출물 작성 및 버전 관리 |

| 회의 종류 | 시점 | 주요 내용 | 사용 방식 및 도구 |
|---|---|---|---|
| Sprint Planning Meeting | Sprint 시작 전 | 작업 범위, 우선순위, 담당 역할 결정 | 오프라인 회의 또는 온라인 회의, GitHub Issue |
| Daily Scrum Meeting | 개발 중 주기적으로 진행 | 진행 상황, 완료 작업, 문제 사항 공유 | 메신저 또는 온라인 회의 |
| Sprint Review | Sprint 종료 시 | 완료된 기능과 산출물 검토 | 온라인 회의 또는 오프라인 회의, GitHub Repository |
| Sprint Retrospective | Sprint 종료 후 | 진행 과정에서의 문제점과 개선 사항 정리 | 온라인 회의 또는 GitHub Issue |
| 비정기 회의 | 문제 발생 시 | 요구사항 변경, 일정 지연, 오류 발생 등 논의 | 메신저, 온라인 회의, GitHub Issue |

## 4.2 변경 사항 관리 방법

개발 과정에서 요구사항이나 기능 범위가 변경될 수 있으므로 변경 사항을 기록하고 관리한다.

변경 사항이 발생하면 변경 내용을 정리하고, 기능, 일정, 품질에 미치는 영향을 검토한 뒤 반영 여부를 결정한다.

변경 사항은 GitHub Issue 또는 커밋 기록을 통해 관리한다.

---

## 4.3 산출물 관리 방법

프로젝트 산출물은 GitHub Repository에서 관리한다.

문서 산출물은 `doc/` 폴더에 저장하고, 파일명은 문서 내용을 쉽게 알 수 있도록 작성한다.

예시는 다음과 같다.

- `project_definition.md`
- `quality_attributes.md`
- `project_management_plan.md`
- `requirements.md`
- `design.md`
- `test_plan.md`

## 4.5 품질 확인 기준

Mini Drive의 주요 기능이 정상적으로 동작하는지 확인하기 위해 기능별 품질 확인 기준을 설정한다.  
각 기능은 정상 동작 여부뿐만 아니라 처리 시간, 정확도, 권한 적용 여부, 오류 상황에서의 처리 방식도 함께 확인한다.

| 확인 항목 | 검사 내용 | 합격 기준 | 불합격 기준 |
|---|---|---|---|
| 파일 업로드/다운로드 | 사용자가 파일을 업로드하고 다시 다운로드할 수 있는지 확인한다. | 100MB 이하의 파일이 10초 이내에 업로드되고, 다운로드한 파일의 내용이 원본과 동일해야 한다. | 파일이 정상적으로 저장되지 않거나, 다운로드한 파일이 원본과 다를 경우 |
| 폴더 관리 | 폴더 생성, 이름 변경, 삭제, 파일 이동이 정상적으로 수행되는지 확인한다. | 폴더 생성, 이름 변경, 삭제, 파일 이동 결과가 3초 이내에 화면과 저장 구조에 반영되어야 한다. | 폴더 구조가 잘못 저장되거나, 파일 이동 결과가 반영되지 않을 경우 |
| 파일 공유 | 특정 사용자에게 파일을 공유하고 권한을 설정할 수 있는지 확인한다. | 보기, 수정, 댓글 권한이 설정 후 즉시 적용되며, 권한이 없는 사용자는 해당 파일에 접근할 수 없어야 한다. | 권한이 없는 사용자가 파일에 접근하거나, 권한 설정이 반영되지 않을 경우 |
| 외부 링크 공유 | 외부 공유 링크 생성과 만료 기간 설정이 정상적으로 동작하는지 확인한다. | 공유 링크가 3초 이내에 생성되고, 설정한 만료 시간이 지난 뒤에는 링크 접근이 차단되어야 한다. | 만료된 링크로 접근이 가능하거나, 링크 권한 검증이 이루어지지 않을 경우 |
| 파일 검색 | 파일 이름, 날짜, 사용자, 파일 유형 기준으로 검색이 가능한지 확인한다. | 검색 결과가 3초 이내에 출력되고, 검색 조건과 일치하는 파일이 95% 이상 표시되어야 한다. | 검색 결과가 누락되거나 조건과 맞지 않는 파일이 표시될 경우, 또는 검색 결과 출력 시간이 3초를 초과할 경우 |
| 파일 버전 관리 | 이전 버전 조회, 다운로드, 복원이 가능한지 확인한다. | 파일 수정 시 이전 버전이 저장되며, 최근 5개 이상의 버전을 조회, 다운로드, 복원할 수 있어야 한다. | 이전 버전이 저장되지 않거나, 복원 결과가 잘못될 경우 |
| 사용자 계정 관리 | 로그인 및 관리자 계정 관리 기능이 정상적으로 동작하는지 확인한다. | 정상 계정은 3초 이내에 로그인되어야 하며, 관리자 권한은 지정된 계정에만 적용되어야 한다. | 잘못된 계정으로 로그인되거나, 관리자 권한이 잘못 적용될 경우 |
| 보안 관리 | 권한 기반 접근 제어와 공유 링크 만료 기능을 확인한다. | 권한이 없는 사용자의 파일 접근은 100% 차단되어야 하며, 만료된 공유 링크는 사용할 수 없어야 한다. | 접근 권한이 없는 사용자가 파일을 열람하거나 수정할 수 있을 경우 |

현재 프로젝트 문서 산출물은 GitHub Repository의 `doc/` 폴더에서 관리한다.

현재 작성되었거나 작성 중인 산출물은 다음과 같다.

| 산출물 | 파일명 | 설명 | 상태 |
|---|---|---|---|
| 프로젝트 정의서 | `project_definition.md` | 프로젝트 명칭, 시스템 설명, 핵심 기능, 유사 소프트웨어 조사 내용을 정리한 문서 | 작성 완료 |
| 품질 요소 정의서 | `quality_attributes.md` | Mini Drive에서 중요하게 고려해야 할 품질 요소를 정리한 문서 | 작성 완료 |
| 프로젝트 관리 계획서 | `project_management_plan.md` | 개발 계획, 품질 관리, 산출물 관리, 리스크 관리 내용을 정리한 문서 | 작성 중 |

현재까지 작성된 문서는 `project_definition.md`, `quality_attributes.md`이며, 본 문서인 `project_management_plan.md`는 추가로 작성하여 같은 폴더에 저장할 예정이다.
